# Envoy Gateway Debug — Troubleshooting Runbook

Troubleshooting for Envoy Gateway (the CNCF Kubernetes Gateway API implementation) and standalone Envoy proxy deployments. Covers Gateway API resources, xDS config, Envoy admin UI, and traffic debugging.

Inspired by: Envoy Gateway docs, Envoy proxy docs, Gateway API SIG docs, solo.io Envoy blog, Tetrate Envoy training.

## When to Activate

Activate when the user asks about:
- Envoy Gateway not routing, HTTPRoute not working
- Gateway API, GatewayClass, Gateway, HTTPRoute
- Envoy admin port 9901, Envoy admin UI
- xDS control plane, Envoy config dump
- Envoy upstream connect error, 503, 502
- Envoy access logs, Envoy response flags
- Envoy rate limiting, circuit breaker
- EnvoyFilter (Istio), custom Envoy config
- Envoy stats, Envoy metrics
- Envoy as standalone ingress (not Istio)

## Troubleshooting Runbook

### Envoy Gateway Architecture (Gateway API)

```
GatewayClass  → defines controller (gateway.envoyproxy.io/gatewayclass)
     ↓
Gateway       → listener config (port, protocol, TLS)
     ↓
HTTPRoute     → routing rules (host, path, headers → backend)
  GRPCRoute   → gRPC routing
  TCPRoute    → L4 TCP routing
     ↓
Envoy Pod     → generated from xDS by control plane
     ↓
Backend Service
```

### Step 1 — Gateway API Resource Status

```bash
# GatewayClass (cluster-scoped)
kubectl get gatewayclass

# Gateways
kubectl get gateway -A
kubectl describe gateway <name> -n <namespace>
# Look for: status.conditions — Ready=True, Programmed=True

# HTTPRoutes
kubectl get httproute -A
kubectl describe httproute <name> -n <namespace>
# Look for: status.parents.conditions — Accepted=True, ResolvedRefs=True

# GRPCRoutes
kubectl get grpcroute -A

# Check all Gateway API resources at once
kubectl get gateways,httproutes,grpcroutes,tcproutes -A
```

## Failure Mode: HTTPRoute Not Routing

**Symptom:** Requests to gateway return 404 or 503

```bash
# Step 1: Verify Gateway is programmed
kubectl get gateway <name> -n <ns> \
  -o jsonpath='{.status.conditions}'
# Programmed=True means Envoy config was successfully applied

# Step 2: Verify HTTPRoute is accepted
kubectl get httproute <name> -n <ns> \
  -o jsonpath='{.status.parents[*].conditions}'

# Step 3: Verify backend service exists and has endpoints
kubectl get svc <backend-svc> -n <ns>
kubectl get endpoints <backend-svc> -n <ns>

# Step 4: Check Envoy Gateway controller logs
kubectl logs -n envoy-gateway-system \
  deployment/envoy-gateway --tail=30

# Step 5: Check the Envoy proxy pod generated by Gateway
kubectl get pods -n <gateway-ns> | grep "envoy-"
kubectl logs -n <gateway-ns> <envoy-pod> --tail=30

# Step 6: Check Gateway address
kubectl get gateway <name> -n <ns> \
  -o jsonpath='{.status.addresses}'

# Test with correct Host header
GW_IP=$(kubectl get gateway <name> -n <ns> \
  -o jsonpath='{.status.addresses[0].value}')
curl -H "Host: myapp.example.com" http://$GW_IP/api

# Common issues:
# hostname mismatch → HTTPRoute hostname doesn't match request Host header
# namespace mismatch → HTTPRoute in different ns than Gateway (needs ReferenceGrant)
# parentRef wrong → HTTPRoute.spec.parentRefs points to wrong gateway
```

**ReferenceGrant (cross-namespace routing):**
```yaml
# Allow Gateway in gateway-ns to reference Service in app-ns
apiVersion: gateway.networking.k8s.io/v1beta1
kind: ReferenceGrant
metadata:
  name: allow-gateway
  namespace: app-ns        # namespace of the target resource
spec:
  from:
  - group: gateway.networking.k8s.io
    kind: HTTPRoute
    namespace: gateway-ns  # namespace of the source
  to:
  - group: ""
    kind: Service
```

## Envoy Admin Interface

Every Envoy proxy exposes an admin API on port 9901. This is the most powerful debugging tool.

```bash
# Port-forward to Envoy admin (works for Envoy Gateway, Istio sidecars, standalone)
kubectl port-forward -n <ns> <envoy-pod> 9901:9901

# In another terminal:
# Health check
curl http://localhost:9901/healthcheck/ok

# Complete config dump (clusters, listeners, routes, endpoints)
curl http://localhost:9901/config_dump | jq .

# Just clusters (upstream services)
curl http://localhost:9901/config_dump | jq '.configs[] | select(.["@type"] | contains("Clusters"))'

# Just routes
curl http://localhost:9901/config_dump | jq '.configs[] | select(.["@type"] | contains("Routes"))'

# Live stats (thousands of metrics)
curl http://localhost:9901/stats

# Filter stats
curl http://localhost:9901/stats?filter=upstream_cx|downstream_cx

# Request counters per cluster
curl http://localhost:9901/stats | grep "cluster\.<name>.*rq_total"

# Reset counters (testing)
curl -X POST http://localhost:9901/reset_counters

# Hot restart, drain (for graceful shutdown testing)
curl -X POST http://localhost:9901/healthcheck/fail
curl -X POST http://localhost:9901/healthcheck/ok

# Logging level (change at runtime)
curl -X POST http://localhost:9901/logging?level=debug
curl -X POST http://localhost:9901/logging?level=info

# List all endpoints and their health
curl http://localhost:9901/clusters | grep -E "^[a-z].*::[0-9].*::"
```

## Envoy Upstream Error Codes

```bash
# Envoy access log response flags:
# UF  = Upstream Connection Failure
# UH  = No Healthy Upstream (all pods unhealthy)
# UC  = Upstream Connection Termination
# UT  = Upstream Request Timeout
# UO  = Upstream Overflow (circuit breaker)
# URX = Upstream Retry Limit Exceeded
# NR  = No Route Match (404 essentially)
# NC  = Upstream Remote Reset
# DPE = Downstream Protocol Error
# -   = No flags (success)

# Stream Envoy access logs
kubectl logs <envoy-pod> -n <ns> -f | jq -r '
  .start_time + " " + .request.method + " " + .request.path +
  " → " + (.response.code | tostring) +
  " [" + .response.flags + "]" +
  " upstream=" + (.upstream.host // "none")'

# Count 503 errors by upstream
kubectl logs <envoy-pod> -n <ns> | jq -r 'select(.response.code==503) | .upstream.cluster' | \
  sort | uniq -c | sort -rn
```

## Envoy Rate Limiting

```bash
# Check rate limit service
kubectl get pods -A | grep ratelimit

# Check HTTPRoute rate limiting (Gateway API)
kubectl get httproute <name> -n <ns> -o yaml | grep -A10 "rateLimit"

# Envoy local rate limit (without external service)
# EnvoyFilter (Istio) or direct Envoy config for LocalRateLimit filter

# Check rate limit stats
curl http://localhost:9901/stats | grep "ratelimit\|rate_limit"

# Common: global rate limit with Redis backend
kubectl get svc -A | grep ratelimit
kubectl logs deployment/ratelimit -n ratelimit-ns --tail=20
```

## Standalone Envoy (Not Istio, Not Envoy Gateway)

Some teams deploy Envoy directly as an ingress without a service mesh.

```bash
# Check Envoy bootstrap config
kubectl get configmap <envoy-config> -n <ns> -o yaml
# Look for: static_resources → clusters and listeners

# Check Envoy version
kubectl exec <envoy-pod> -n <ns> -- envoy --version

# Envoy hot restart (zero-downtime config reload)
# Signal: SIGUSR1 → increments epoch
# Or via admin: curl -X POST http://localhost:9901/reset_counters

# xDS dynamic config via control plane
# Check connection to xDS server:
curl http://localhost:9901/config_dump | \
  jq '.configs[] | select(.["@type"] | contains("DynamicResources")) | .dynamic_warming_clusters'
```

## EnvoyFilter (Istio Extension)

EnvoyFilter modifies Istio's generated Envoy config — powerful but fragile.

```bash
# List EnvoyFilters
kubectl get envoyfilter -A

# Validate EnvoyFilter
kubectl describe envoyfilter <name> -n <ns>

# Check if EnvoyFilter is causing issues
kubectl get pod <pod> -n <ns> -o yaml | grep "istio.io/rev"
istioctl proxy-config listeners <pod> -n <ns> | grep -A5 "custom"

# Common issues:
# patch context wrong (sidecar vs gateway)
# applyTo wrong (CLUSTER vs LISTENER vs HTTP_FILTER)
# match wrong (port, name, filter chain)

# Remove problematic EnvoyFilter and test
kubectl delete envoyfilter <name> -n <ns>
```
